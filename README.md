# Advanced B-Tree (2-3-4 Tree) Assignment

A full C++ implementation of a **2-3-4 Tree** (B-Tree of minimum degree `t = 2`) with dynamic operations, persistence, logging, and structural analysis.

## Features
- Top-down **proactive-splitting** insertion (CLRS algorithm)
- **Deletion** with borrowing and merging
- **Search**, in-order **traverse**, and **level-order** (BFS) printing
- **Split counter** (static `BTreeNode::splitCount`)
- **File-driven** operation parsing from `input.txt`
- **Per-operation logging** to `log.txt`
- **Persistence**: `SAVE` / `RESTORE` via `snapshot.dat` (pre-order serialization with leaf-flag markers)
- Invalid lines are safely skipped
- **No STL containers** – only raw arrays and pointers

## Operation File Format (`input.txt`)
| Syntax        | Meaning                                    |
|---------------|--------------------------------------------|
| `I x`         | Insert integer `x`                         |
| `D x`         | Delete integer `x` (ignored if absent)     |
| `S x`         | Search for `x` (prints `FOUND` / `NOT FOUND`) |
| `T`           | Print in-order traversal                   |
| `L`           | Print level-order                          |
| `SAVE`        | Serialize the tree to `snapshot.dat`       |
| `RESTORE`     | Deserialize the tree from `snapshot.dat`   |
| `CLEAR`       | Empty the tree                             |
| `# comment`   | Ignored                                    |
| anything else | Logged as invalid and skipped              |

## Serialization Format (`snapshot.dat`)
Each line describes one node in **pre-order**:
```
<leaf_flag> <n> <k1> <k2> ... <kn>
```
Non-leaf nodes are followed immediately by their `n + 1` children, recursively. The `leaf_flag` lets the parser know whether to recurse.

### README Questions

**1. Why is saving only keys not enough?**
The same multiset of keys can produce many different tree structures depending on insertion order. Persisting just the keys loses:
- which keys share a node,
- parent-child relationships,
- the `leaf` flag of each node.
Without these, reconstruction would require re-inserting and would almost certainly yield a different tree.

**2. How do you distinguish node types in serialization?**
Every serialized record begins with a **leaf flag** (`1` for leaf, `0` for internal). When the parser reads a node with `leaf_flag == 0`, it knows to recursively deserialize the next `n + 1` children; otherwise it stops at that node. This simple marker is enough to disambiguate structure.

## Complexity
| Operation | Time        | Space  |
|-----------|-------------|--------|
| Search    | O(log n)    | O(1)   |
| Insert    | O(log n)    | O(1) amortized per split |
| Delete    | O(log n)    | O(1) |
| Traverse  | O(n)        | O(h) recursion |
| Level-order | O(n)      | O(n) queue |
